- Add persisted records:
  - `ItemCategoryDefinition`: `id`, `name`.
  - `ItemSubcategoryDefinition`: `id`, `categoryId`, `name`, `mechanicalRole`, `allowedEquipSlots[]`, `occupiedSlots[]`.
  - `mechanicalRole` is a fixed enum: `melee | range | shield | body_armor | occult | consumable | accessory`.
- Keep current seeded taxonomy, but express it as persisted defaults:
  - Categories remain the current groups (`melee`, `body_armor`, `shield`, `range`, `occult`, `neck`, `rings`, `head`, `orbital`, `charm`, `consumable`).
  - Accessory-like groups map to `mechanicalRole: accessory`; combat-facing groups keep their current combat roles.
- Change blueprint persistence:
  - `ItemBlueprintRecord` gains `categoryDefinitionId` and `subcategoryDefinitionId`.
  - Legacy `category` / `subtype` inputs remain readable during hydration, but new writes resolve through definition IDs.
  - `SharedItemRecord` should no longer be authoritative for category/subtype; runtime resolves behavior from the item’s blueprint and its linked definitions.
- Add centralized definition-driven helpers and move all equip/combat/summary code to them:
  - Resolve category/subcategory records from blueprint.
  - Resolve effective equip behavior from `mechanicalRole`, `allowedEquipSlots[]`, `occupiedSlots[]`, and `combatSpec`.
  - Replace literal checks like `item.category === "shield"` or `item.subtype === "earring"` across item summaries, equip validation, default slot choice, loadout rendering, and combat weapon selection.
- Define the temporary occupancy rule for this refactor:
  - `allowedEquipSlots[]` is the set of valid anchor choices.
  - `occupiedSlots[] = []` means “occupy only the resolved anchor slot” for now.
  - Non-empty `occupiedSlots[]` means a fixed occupied-slot set; this is enough to preserve current two-hand and shield behavior until `ITEM-MULTISLOT-01` adds explicit anchor persistence.
- Preserve current behavior defaults through definitions:
  - One-handed melee/ranged/occult items stay primary-first, then secondary if free.
  - Shields resolve to secondary hand only.
  - Rings resolve left, then right.
  - Body/neck/head stay single-slot.
  - Supplementary `orbital`, `earring`, and `charm` definitions exist, but player-facing activation remains deferred exactly as today.
- Persistence and migration:
  - Extend app flow state and storage envelope with `itemCategoryDefinitions` and `itemSubcategoryDefinitions`.
  - Seed defaults when absent.
  - Migrate existing blueprints by mapping current `category` / `subtype` to seeded definition IDs.
  - Continue reading legacy item records and legacy blueprint aliases exactly as today, but normalize them onto the new definition-backed blueprint shape during hydration.
  - Starter items remain blueprint-backed and must also be definition-backed after hydration.
- DM tooling:
  - Add a dedicated DM definition management page for categories and subcategories.
  - Blueprint management switches from enum-backed category/subtype selects to persisted definition selects.
  - Subcategory options are filtered by selected category.
  - Deleting a category is blocked when it still has subcategories or referenced blueprints.
  - Deleting a subcategory is blocked when any blueprint references it.
  - Renames and safe mechanical edits propagate automatically because blueprints/items resolve by definition ID, not copied labels.
- Combat and derived runtime:
  - `combatEncounterPhysicalAttacks` must resolve attack profiles from definition-backed role + combat spec, not literal category/subtype strings.
  - Item modifier application remains based on equipped/active item IDs exactly as now; only equip eligibility and summary/combat classification change.

## Public Interface Changes
- `AppFlowContextValue` and persisted app state gain `itemCategoryDefinitions`, `itemSubcategoryDefinitions`, plus CRUD methods for both.
- `ItemBlueprintRecord` gains `categoryDefinitionId` and `subcategoryDefinitionId`.
- New shared helpers become the only supported way to answer:
  - what slot(s) an item can use,
  - what slot(s) it occupies,
  - whether it is a combat weapon, shield, accessory, armor, occult focus, or consumable,
  - what summary label to show.
- Legacy `category` / `subtype` string fields remain read-compatible only; new logic must not branch on them.

  - empty-state seeding of definitions,
  - serialization/deserialization of definition arrays,
  - migration of current saves with blueprint `category` / `subtype`,
  - continued support for legacy item/blueprint aliases,
  - starter items still resolving to valid blueprints and definitions.
- Extend item behavior tests to cover:
  - definition-driven shield secondary-hand enforcement,
  - one-handed weapon coexistence with shield,
  - ring left/right default placement,
  - summary labels coming from definition slots instead of hardcoded category checks.
- Extend physical attack tests to cover:
  - melee/ranged profile selection from definition-backed mechanical role,
  - unchanged outcomes for unarmed, brawl, one-handed, two-handed, oversized, and ranged defaults after migration.
- Run full validation after the refactor group:
  - `npm run typecheck`
  - `npm test`
  - `npm run build`

## Assumptions And Defaults

- Chosen safety policy: block deletes for referenced definitions; allow renames and safe mechanical edits with automatic propagation.
- `ItemCategoryDefinition` is a grouping record; `ItemSubcategoryDefinition` is the effective behavior record.

- `occupiedSlots[] = []` is the temporary anchor-only convention for single-slot items until `ITEM-MULTISLOT-01` introduces explicit anchor persistence.
- Supplementary slot visibility/activation remains deferred; this refactor only prepares the data model and rule resolution for it.
